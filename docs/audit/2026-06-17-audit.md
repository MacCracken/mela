# mela security audit — 2026-06-17 (v0.9.0)

> Pre-release audit of the Cyrius port. mela is a **supply-chain trust boundary**
> (it decides what gets installed), so the bar is high. This audit maps current
> external 0-day / CVE classes onto mela's actual attack surface, records the
> control for each, and lists the hardening applied this release. Companion:
> [`../development/threat-model.md`](../development/threat-model.md).

## Scope

The ported modules and their external-data entry points:

- **trust** — Ed25519 verify, SHA-256 (via `sigil`).
- **transparency** — append-only hash-chained log + JSON import.
- **manifest / category** — JSON manifest codec.
- **local_registry** — `index.json` load, install (name → install dir).
- **remote_client** — URL building, response JSON, response cache (transport is a
  deferred local seam, ADR-0006 — not yet a live network parser).
- **sandbox_profiles / ratings** — JSON import, ratings store JSON.
- **flutter_agpkg** — **gzip + hand-rolled ustar** pack/inspect (the highest-risk
  surface: a hand-written archive parser over attacker-supplied bytes).

## Findings

Severity is *residual* risk after the listed control.

| # | Threat class (with current CVE/reference) | mela surface | Control | Residual |
|---|---|---|---|---|
| A | **Tar path-traversal / zip-slip / symlink escape** — CVE-2025-45582 (GNU tar two-step symlink), tar-fs CVE-2024-12905, mholt/archiver zip-slip (CVE-2025-3445), pip symlink extraction | hand-rolled ustar reader (`agpkg_inspect` / `agpkg_read_entry`) | **Hardened this release**: `_tar_entry_safe` rejects non-regular typeflags (symlink `2` / hardlink `1` / dir) and any name that is absolute (`/…`) or contains `..`. inspect skips them; read_entry refuses them. mela does not yet extract to disk — when it does, the same guard gates each write. | **Low** |
| B | **Decompression bomb (gzip bomb)** — CVE-2026-49853 (Tornado unbounded gzip), the 42.zip class | `gzip_decompress` of a downloaded bundle | **Bounded output**: decompression targets a fixed `AGPKG_BUF_MAXLEN` (8 MiB) buffer; `sankoch` writes at most `dst_cap` and errors past it, so a bomb cannot exhaust memory — it fails closed. | **Low** |
| C | **Ed25519 signature malleability / non-canonical S** — RFC 8032 §5.1.7 (reject S ≥ L); OpenSSL #7693 (1.1.1 accepted malleable sigs) | `trust_verify` / `registry_verify_package` | `sigil`'s `ed25519_verify` enforces the §5.1.7 `S < L` check (rejects non-canonical S → non-malleable) and canonical point decoding; mela additionally **rejects any signature not exactly 64 bytes** before calling verify. Verify is keyed by the manifest's publisher `key_id` resolved against the keyring (no bare-key trust). | **Low** |
| D | **Signature stripping / unsigned artifact** — supply-chain "unsigned/unverified artifacts" (OWASP 2025 A03) | `pipeline_install` | **Gate 1 enforced**: install refuses a bundle whose publisher `key_id` has no current keyring key, or whose signature fails to verify over the bundle bytes. An unsigned or sig-stripped bundle installs nothing. Tested (`pipeline`, `registry-signature`). | **Low** |
| E | **Tampered artifact in transit (integrity)** — registry/CDN MITM, "build pipelines without integrity verification" | `pipeline_install` gate 2 | **Gate 2 enforced**: the bundle's SHA-256 must equal the expected digest (download integrity); a mismatch installs nothing. The transparency log records the content + signature hashes of every publication (append-only, hash-chained, tamper-evident). Tested. | **Low** |
| F | **Malicious package name → path traversal at install** | `registry_install` install dir = `packages/<name>` | The manifest name is validated (`manifest_validate`: lowercase `[a-z0-9-]` only) **before** install, so `..` / `/` / NUL cannot appear in the install path. Enforced in both `registry_install` and `pipeline_install`. | **Low** |
| G | **Typosquatting** — look-alike package names | name-based discovery | Trust is **key-based, not name-based**: a look-alike package carries a different publisher `key_id`, which must already be in the keyring to install. Names alone confer no trust. Defense is operational (curate the keyring); mela provides the key gate. | **Medium** (operational) |
| H | **Dependency confusion** — public name shadows a private one with a higher version | `DependencyGraph` resolver | mela resolves a **single** explicit graph (`add` + `resolve`); there is no public/private split or "highest-version-wins across registries" resolution to confuse. `check_missing` surfaces unresolved deps rather than silently substituting. | **Low** |
| I | **Malformed external data → crash / OOB** — parser 0-days generally | every JSON parser, the ustar reader | Every external-data parser is **fuzzed** (`tests/mela.fcyr`: manifest / transparency / registry / remote / sandbox / ratings JSON + the gzip+ustar inspector) and returns an error sentinel (0) on garbage rather than faulting. The ustar reader is bounds-checked (`off + 512 ≤ dlen`, size capped, name ≤ 100). | **Low** |
| J | **Expired / rotated publisher key accepted** | keyring | `KeyVersion` carries a validity window; `keyring_get_current_key(key_id, now)` returns only a key valid at `now`, so an expired key is not used to verify. Tested (`keyversion` / `keyring`). | **Low** |

## Hardening applied this release

- **`_tar_entry_safe`** (finding A) — the tar reader now rejects symlink/hardlink/
  non-regular entries and traversal/absolute names; `agpkg_inspect` skips them and
  `agpkg_read_entry` refuses them. New `hardening` test group proves a `../evil.so`
  and `/etc/passwd` entry are skipped while `bin/safe.so` passes.

## Residual risk & follow-ups (tracked for v0.9.1 / v1.0)

- **F→extraction**: when on-disk tarball extraction lands (deferred, ADR-0005),
  it must apply `_tar_entry_safe` per write *and* canonicalize the resolved path
  stays within the destination (the CVE-2025-45582 two-step symlink case).
- **Transport (ADR-0006)**: a live HTTPS client is not yet wired; when it is, TLS
  is stdlib `tls`/`tls_native` (no OpenSSL), and the response body feeds the same
  bounded decompressor + verified install path.
- **Keyring provenance (G)**: document the operational process for populating /
  rotating trusted publisher keys; consider key-pinning guidance for consumers
  (ark) at v1.0.
- **`ed25519_verify` cost** — ~8 ms/op (see benchmarks); not a vulnerability, but
  a DoS consideration for any future server-side bulk-verify path. Track as a
  `sigil` optimisation.
